# The Three Core Components of a Computer System
Any computer system, at the highest level, is built from three main building blocks: the **Processor** (CPU), the **Memory**, and the **I/O** (covered later). These blocks are wired together through three separate buses:
> - **Address Bus:** carries the location (address) the CPU wants to access
> - **Data Bus:** carries the actual data being transferred
> - **Control Bus:** carries control signals such as "read" or "write"

Everything else in the system architecture builds on top of this wiring, so the plan is to take these blocks one at a time, starting with the processor.
# The Processor:
The processor is built from three main pieces:
1. **ALU** (Arithmetic and Logic Unit)
2. **Control Unit**
3. **Registers**

Its entire job, distilled down to its essence, comes down to three main things: **fetch**, **decode**, and **execute** an instruction. This cycle - the **Instruction Life Cycle** - is the beating heart of everything a processor does, repeating continuously no matter how complicated a program looks from the outside.
## The Instruction Life Cycle
## Step 1: Fetch
The code the processor runs doesn't live inside the microprocessor - it lives outside, in **ROM**, which holds the program. 
A special register, the **Program Counter (PC)**, keeps track of which instruction currently has its turn for execution. During fetch, the Control Unit goes out to memory using the address in the PC, pulls that instruction back from ROM, and loads it into a dedicated register called the **IR - Instruction Register**, bringing it "inside" the CPU so it becomes accessible for processing.
> Fetching means retrieving the instruction due for execution from ROM and loading it into the IR so it becomes accessible to the CPU.

Before execution, instructions exist purely as machine language - sequences of 0s and 1s stored in Flash/ROM. This stored program is "the code": a sequence of instructions, one after another, each waiting its turn to be fetched and executed.
## Step 2: Decode
Once the instruction sits in the IR, the Control Unit's decoder circuit figures out what that string of bits actually means - is it addition, multiplication, division, or something else? To decode an instruction, the CPU needs two things:
1. **The Instruction Set**
2. **The Instruction Format**

The **Instruction Format** describes how the microprocessor expects an instruction's bits to be laid out. Take an 8-bit instruction:
$$ 00110110 $$

The format tells the CPU how to split these bits into meaningful chunks - for instance, that the first three bits together mean something specific, which the CPU looks up in a built-in table: the **Instruction Set**.
> **Instruction Set:** A table built into the hardware that maps specific bit patterns to specific operations the processor understands.

Inside that table are two pieces: the **opcode** (the bits representing the operation) and the **operand(s)** (the bits representing the data). Walking through an example instruction:
$$ 0001101101 $$

The CPU pulls out the first three bits (the opcode per the format) and looks them up - say they decode to "ADD." The Control Unit now knows to send a signal to the ALU to perform an addition, feeding it Operand 1 and Operand 2 - for example, $7 + 1 = 8$.
## Step 3: Execute
The Control Unit sends the operation and operands to the ALU, which carries out the operation and produces a result. In older designs, this result went into one specific dedicated register; in modern microprocessors, it's typically stored in any register from the **General Purpose Register file** - a pool of registers set aside for temporary storage.

> **The Full Cycle:**
> 
> 1. **Fetch:** Control Unit retrieves the instruction pointed to by the PC from ROM into the IR.
> 2. **Decode:** The decoder splits the instruction into opcode and operand(s) using the Instruction Format, then looks up the opcode in the Instruction Set.
> 3. **Execute:** The Control Unit signals the ALU with the operation and operands; the result is stored in a register.
## The Program Counter and the Reset Vector
The **Program Counter** always points at the instruction in ROM next in line for execution. On power-on or reset, the PC is set to point at the **Reset Vector**, the first memory location holding executable code, and the true hardware-level starting point of the program, distinct from where `main()` conceptually begins in a high-level language. 
After the PC is aimed at the Reset Vector, a handful of initialization steps run first, and only afterward does control flow to `main`.

As the program runs, the PC keeps **incrementing**, and with every increment the fetch–decode–execute cycle repeats on whatever instruction it now points to.
## Assembling
Writing programs directly in raw binary would be impractical, so programmers use **Assembly language** readable mnemonics instead of raw bits. **Compilers/assemblers** translate these mnemonics into the machine code a specific processor understands.

Different microprocessors can use entirely different mnemonics and syntax for the same conceptual operation (`add` vs. `ADD`, for instance), and different processors may natively understand different sets of operations altogether — one might support only addition and subtraction, while another also supports multiplication. If you try to run a multiplication on hardware that only understands addition, _something_ has to convert that multiplication into a series of additions. That something is the **compiler**.

> **Two systems work together to run a program correctly:
> 
> - **Hardware**:
> 	- The Instruction Decoder, which actually executes the instruction set the processor was built for.
> 	
> - **Software**:
> 	- The Compiler, which can translate unsupported operations (like multiplication) into sequences of supported ones (like repeated addition) when the hardware can't do it natively.

## RISC vs. CISC
This leads to **ISA** - Instruction Set Architecture - and two key terms:
- **RISC**: Reduced Instruction Set Computer (few instructions)
- **CISC**: Complex Instruction Set Computer (many instructions)

Because RISC hardware understands relatively few operations, its compiler has to be **powerful**, doing the heavy lifting of decomposing complex operations into the reduced set the hardware supports. CISC hardware already understands many operations directly, so its compiler can stay comparatively **simple**.
## Hardwired vs. Microprogrammed Control
There are two manufacturing approaches for the circuitry that decodes instructions:
- **Hardwired Control**: built from a fixed collection of logic gates wired directly together. It's **fast**, since signals flow straight through with no lookup step. But supporting more instructions requires more gates, which quickly becomes expensive. This is why **RISC** favors hardwired control: few instructions, few gates, fast, cheap.
- **Microprogrammed Control**: works more like a search process, looking up the correct sequence of micro-operations for a given instruction. It's slower than hardwired logic, but it can represent a much larger instruction set without a proportional explosion in gate count, since new instructions are added as lookup-table entries rather than physical gates. This is why **CISC**, which needs to support many instructions, leans on microprogrammed control - high compiler simplicity, at the cost of more complex hardware.
### Comparing RISC and CISC

| Dimension               | RISC                  | CISC                        |
| ----------------------- | --------------------- | --------------------------- |
| Instruction count       | Few                   | Many                        |
| Compiler                | Powerful, expensive   | Simple, cheap               |
| Hardware                | Simple, cheap, small  | Complex, expensive, large   |
| Control style           | Hardwired (typically) | Microprogrammed (typically) |
| Decode speed (isolated) | Faster (direct gates) | Slower (lookup process)     |
| Power consumption       | Lower                 | Higher                      |
On **overall system performance**, the two roughly balance out: CISC "loses" time in hardware lookup, while RISC "loses" time in the compiler's translation overhead - add both hardware and software costs together, and the two land at roughly the same place.

### When Do You Pick One Over the Other?
If a system genuinely needs a rich, expansive native instruction set, CISC is the natural fit. If a system only needs a small, focused set of operations and doesn't want to pay the cost (in gates, silicon area, power) of instructions it will never use, RISC is the sensible choice — you only pay for what you actually use. This is the underlying philosophy behind why embedded systems, including the microcontrollers this course builds toward, lean on RISC-style architecture.

## Registers:
Registers store data **temporarily** - each with its own dedicated job (holding the fetched instruction, holding a recent ALU result, and so on). A result placed in a register isn't its permanent home; it eventually has to make its way back to memory over the buses.

Inside the CPU, registers are referenced by **name**, not address — for example, `$1` for Register 1, with the dollar sign signaling "this is a register name." Outside the CPU, memory locations are referenced by **address**. Mixing the two up in assembly code (using an address where a register name is expected) produces an error, because the CPU expects a name, not an address, at that position.

> **Key Point:** Inside the CPU, registers are referenced by **name**. Outside, in memory, locations are referenced by **address**. The two cannot be used interchangeably.
### The Flags Register
The **Flags register** (status register) has individual bits, each called a **flag**, tracking a specific fact about the most recent ALU operation.
- **Sign Flag:** reports whether the last result was negative. A `1` means the result was negative; `0` means it wasn't.
- **Zero Flag:** reports whether the last result was exactly zero.
- **Overflow Flag:** tracks whether a result exceeded the representable range.

Not every instruction updates the flags - only instructions that actually pass through the ALU do. A **jump** instruction, for example, never touches the ALU (it just redirects execution flow), so it leaves every flag untouched. An **increment** instruction, which some microprocessors internally convert into "add 1," genuinely does route through the ALU, so it _does_ update the flags.
>  Flags only change when an instruction causes the ALU to perform an operation. Control-flow instructions like jumps leave every flag exactly as it was.
