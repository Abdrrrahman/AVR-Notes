# Dynamic Memory
The biggest difference between static memory and dynamic memory is that static memory is constant during run-time, while dynamic memory can grow and shrink during run-time.
### `malloc()`
`void * malloc(size_t)` Memory Allocation Standard Library Function is used to allocate some memory locations. It takes the number of bytes to be allocated during run time as an argument. It allocates these memory locations at the heap memory segment inside the RAM. It returns the address of the first byte allocated. The return type is `void *` — The C's way to return something generic — which can be casted into any type you want.
```c
// Code Snippet 1-1

#include <stdlib.h>
```

The function is defined inside the standard library.
```c
// Code Snippet 1-2

short *p = malloc(100);
p++; // p now is 102
```

Here it returns a pointer pointing to the first byte address was allocated, when we incremented the pointer, it is incremented by a whole `short` step — 2 bytes — to become 102. If it failed allocating, it returns `nullptr`. You can call it with an `rvalue` like we did above, or you can call it by `lvalue` like this:
```c
// Code Snippet 1-3

#include <stdlib.h>
#include <stdio.h>

int main(void) {
	long bytesNum;
	scanf("%i", &bytesNum);
	long *p = (long *) malloc(bytesNum);
	
	return 0;
}
```

Here what happens is that I am calling it using another variable `lvalue`, so it works fine. Here I casted it from a `void *` into `long *` so the compiler doesn't warn.

Imagine the following scenario: You want to make an application with dynamic array size, the size of the array changes during run time according to the user input, then takes elements to that array then calculate their sum.

### `calloc()`
`void * calloc(size_t, size_t)` Contiguous Allocation Standard Library Function is used to allocate space that is formatted for multiple elements like arrays or data structures. It takes two arguments: one is for the number of elements to be allocated, the other is the size of each element in bytes. It multiplies them together then it allocates the memory needed. It returns `void *` to be generic also. After creating the memory allocations it zeroes them (or we can say it defaults them).
```c
// Code Snippet 1-5

#include <stdio.h>
#include <stdlib.h>

int main(void) {
	long *calloc_ptr = (long *) calloc(2, sizeof(long));
	long *malloc_ptr = (long *) malloc(sizeof(long));
	
	printf("The Calloc Pointer is = %i.\nThe Malloc Pointer is = %i.\n", *calloc_ptr, *malloc_ptr); 
	
	return 0;
}
```

The result will be:
```bash
# Bash Snippet 1-1

The Calloc Pointer is = 0.
The Malloc Pointer is = 7667904. # Garbage value
```
### `realloc()`
`void *realloc(void *, size_t);` Memory Reallocation Standard Library Function is used to reallocate some memory addresses that have been allocated earlier. You can make it bigger or smaller. It takes two arguments: a pointer to the address you want to reallocate, and the new number of bytes you want to allocate from this address. It returns `void *` for generic purposes and it needs to be casted.

It doesn't initialize memory into zeros like `calloc` does.

**One Possibility is when you send a number of bytes that is less than the allocated before:** This means that you are freeing what is excess. As an example, if you had allocated 100 bytes using `malloc`, you can reallocate only 70 of them and free the other 30. You have to assign the `realloc` function to another pointer.

**Another Possibility is when you send a number of bytes that is greater than the allocated before:** This means that you are trying to allocate some new memory locations in addition to those you have allocated before. As an example, if you had allocated 100 bytes using `malloc`, and you want to add 50 more bytes to them, you can reallocate the same 100 bytes and add to them 50 more. But unfortunately, we cannot always find a free memory location beneath us. If we are lucky, we can find that there are 50 free memory locations beneath us. If we are not, the OS searches for an empty slot that this program can have its 150-contiguous bytes. If found, it copies all of its content in the 100 bytes — the already allocated locations — into the new memory location with the 150-byte unallocated space. If not found, the function returns `nullptr`.
#### Example:
```c
// Code Snippet 1-6

#include <stdio.h>
#include <stdlib.h>

int main(void) {
	long *p1 = (long *) malloc(100);
	long *p2 = (long *) realloc(p1, 150);
	
	printf("Address before is %i\nAddress after is %i\n", p1, p2);
	
	return 0;
}
```

Output:
```bash
# Bash Snippet 1-2

Address before is 11215152 
Address after is 11215152
```

You can notice here that the two addresses are the same, as the OS successfully found 50 more bytes contiguous with the 100 before.
#### Example:
```c
// Code Snippet 1-7

#include <stdio.h>
#include <stdlib.h>

int main(void) {
	long *p1 = (long *) malloc(100);
	long *p2 = (long *) realloc(p1, 20000);
	
	printf("Address before is %i\nAddress after is %i\n", p1, p2);
	
	return 0;
}
```

Output:
```bash
# Bash Snippet 1-3

Address before is 10887472
Address after is 10887504
```
You can notice here that the two addresses are different, which means that the function has found not enough contiguous memory starting from `10887472` that can hold another `20000` bytes, so it searched for another place, then copied the contents of the first 100 bytes to the new address and allocated the remaining bytes.
#### Example:
```c
// Code Snippet 1-8

#include <stdio.h>
#include <stdlib.h>

int main(void) {
	long *p1 = (long *) malloc(100);
	long *p2 = (long *) realloc(p1, 20000000);
	
	printf("Address before is %i\nAddress after is %i\n", p1, p2);
	
	return 0;
}
```

Output:
```bash
# Bash Snippet 1-4

10887472
0
```
You can notice here that it returned null, which means that it has found no place that can hold `20000000` contiguous bytes in the heap.
### `calloc` vs `malloc`
Both allocate memory in contiguous blocks, although `calloc` is explicitly defined as Contiguous Memory Allocation Standard Library. By convention it is called Clear Allocation.
#### Why `calloc` Has "Contiguous" in Its Name
If both functions allocate contiguous space, why did the creators of C put "Contiguous" in `calloc`'s name?

1. **Array-Centric Purpose:** `calloc` was specifically designed to allocate memory for _arrays_ of objects. Its arguments force you to specify the number of elements and the size of each element (`calloc(num, size)`), conceptually emphasizing a continuous sequence of identical elements.
2. **Historical Context:** In early computer systems, `calloc` was often optimized to grab large, contiguous pages of physical memory directly from the operating system and zero them out simultaneously, distinguishing its intended use case from smaller, generic `malloc` requests.    

The biggest difference between them is that `calloc` initializes the allocated memory to zero, while `malloc` leaves the memory uninitialized (containing random "garbage" values). This makes `malloc` faster but it requires manual calculation by humans and leaves garbage values, while `calloc` is slower as it requires initializing the memory into zero but does the calculation automatically.
- Use **`calloc`** when you need a clean slate (like initializing arrays, counters, or structures to zero) and want to avoid accidental bugs from garbage data.
- Use **`malloc`** when you are going to immediately overwrite the memory anyway (like copying a string or filling an array from a file), as it avoids the performance cost of clearing the memory twice.

### `free`
`void free(void *)` The free function is used to unallocate the allocated memory using `malloc`, `calloc`, and `realloc`. It is essential in every program that has an allocated memory as it prevents memory leaks.
```c
// Code Snippet 1-9

#include <stdio.h>
#include <stdlib.h>

int main(void) {
	long *ptr = (long *) calloc(2, sizeof(long));	
	free(ptr);
	return 0;
}
```
Local variables are called automatic variables as they get allocated and unallocated (freed) automatically; they do not need `free` to release the memory.
### Problems with Dynamic Memory
#### Non-Determinism:
If you are using dynamic memory, you can encounter a problem with allocating; it can return `nullptr` if it failed. This is vague; in embedded systems this is unacceptable. In Embedded Systems you must be aware of the space and time needed for a specific program before run time.
#### Fragmentation:
This is the main problem that makes dynamic memory restricted — not favored — in embedded systems. Memory fragmentation means that the memory is divided.
Imagine you have a 10 Kbytes Heap. At first, you allocated 3 Kbytes. After some time you allocated another 4 Kbytes. After some time you freed the 3 Kbytes you allocated before. Now you have — from top to down — 3 Kbytes free, 4 Kbytes allocated, and another 3 Kbytes free, which means you have 6 Kbytes free on your heap. However, you cannot allocate a block more than 3 Kbytes. If you tried to allocate a block of 5 Kbytes, which is hypothetically allowed, you will get a `nullptr` return value because you do not have 5 contiguous Kbytes, even though you have 6 Kbytes of total free memory.

A solution can be to shift the in-between 4 Kbytes memory somewhere else, but this is also inconvenient in embedded systems as you are wasting run time and lowering performance.
```c
// Code Snippet 1-10

#include <stdio.h>
#include <stdlib.h>

#define K (1024)

int main(void) {
	char *p1 = (char *) malloc(3*K);
	char *p2 = (char *) malloc(4*K);
	
	free(p1);
	
	p1 = malloc(4*K);
	// Returns Null on a system that has a 10 KB heap memory
	// This is called Memory Fragmentation
	return 0;
}
```
