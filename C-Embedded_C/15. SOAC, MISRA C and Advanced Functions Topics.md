# SOAC Rule
Spiral Outwards Anticlockwise is a rule to read variables' names.

The rule is simple: you move clockwise/anticlockwise depending on the scenario (often formally known as the **Right-Left Rule**). If you hit a `()` that is not a function call, then it has the highest precedence, and `[]` subscript operators have higher precedence than `*` pointers.
### If you come across:
- `(datatype)`: It is interpreted as `Function(s) take(s) datatype and return(s)`.
- `[size]`: It is interpreted as `Array(s) of size`.
- `*ptr`: It is interpreted as `Pointer(s) to`.

# MISRA C

**MISRA C** (Motor Industry Software Reliability Association C) is a set of coding guidelines designed to ensure that C code is **safe, reliable, and portable** — particularly in safety-critical embedded systems.
### Who Uses It?
MISRA C is **mandatory in the automotive embedded industry**. It is also commonly adopted in:
- **Aerospace**
- **Railway systems**
- **Nuclear control systems**

> It is _not_ required for general electrical or medical embedded systems, though some projects choose to adopt it voluntarily.
### Tooling
Many IDEs and static analysis tools have MISRA C integration that flags violations automatically. Examples include **PC-lint**, **LDRA**, **Polyspace**, and **Parasoft C/C++test**.
## Rule Categories
MISRA C rules fall into two categories:

|**Category**|**Description**|
|---|---|
|**Required (Mandatory)**|Must be followed; violations are not acceptable|
|**Advisory**|Strongly recommended; deviations require documented justification|

## Rules Reference
### 1. No Unreachable Code
**Category:** Required

Unreachable code is any code that can **never be executed** due to control flow — typically code placed after a `return`, `break`, or similar exit point.

> **Unreachable code** -> will never execute at all.
> **Dead code** -> may execute but has no effect on program behavior.

Both are prohibited, but dead code is considered worse: it wastes flash memory, execution memory, _and_ execution time with zero benefit. Unreachable code, if not eliminated by the compiler optimizer, wastes only flash memory.
```c
// Code Snippet 1-1

// MISRA VIOLATION
int foo(int x, int y) {
    int diff = x - y;  // Dead code: computed but never used
    return x + y;
    int sum = x + y;   // Unreachable code: after return
}

// MISRA COMPLIANT
int foo(int x, int y) {
    return x + y;
}
```
### 2. No `goto` Statement
**Category:** Required

The `goto` statement creates arbitrary jumps in control flow, making code difficult to reason about, audit, and test.
```c
// Code Snippet 1-2

// MISRA VIOLATION
void process(void) {
    goto cleanup;
    do_work();
cleanup:
    free_resources();
}

// MISRA COMPLIANT — use structured control flow
void process(void) {
    do_work();
    free_resources();
}
```

### 3. No `continue` Statement
**Category:** Required

`continue` skips the rest of a loop body and jumps to the next iteration. This creates hidden control flow paths that are easy to miss during code review.
```c
// Code Snippet 1-3

// MISRA VIOLATION
for (i = 0; i < N; i++) {
    if (arr[i] < 0) {
        continue;  // Hidden jump
    }
    process(arr[i]);
}

// MISRA COMPLIANT
for (i = 0; i < N; i++) {
    if (arr[i] >= 0) {
        process(arr[i]);
    }
}
```

### 4. At Most One `break` per Loop
**Category:** Required

Each iteration statement (`for`, `while`, `do...while`) may contain **at most one `break`** used for loop termination. Multiple `break` statements make it hard to determine when and why a loop exits.
```c
// Code Snippet 1-4

// MISRA VIOLATION
for (i = 0; i < N; i++) {
    if (arr[i] == TARGET) { break; }
    if (arr[i] < 0)       { break; }  // Second break — not allowed
}

// MISRA COMPLIANT — combine exit conditions
for (i = 0; i < N; i++) {
    if ((arr[i] == TARGET) || (arr[i] < 0)) {
        break;
    }
}
```

### 5. Single Point of Exit per Function
**Category:** Required

A function shall have **exactly one `return` statement**, placed at the end. Multiple return points make control flow harder to audit and trace.
```c
// Code Snippet 1-5

// MISRA VIOLATION
int find(int *arr, int n, int target) {
    for (int i = 0; i < n; i++) {
        if (arr[i] == target) {
            return i;   // Early return — not allowed
        }
    }
    return -1;
}

// MISRA COMPLIANT
int find(int *arr, int n, int target) {
    int result = -1;
    for (int i = 0; i < n; i++) {
        if ((arr[i] == target) && (result == -1)) {
            result = i;
        }
    }
    return result;
}
```

### 6. Compound Statements for Loop and `switch` Bodies
**Category:** Required

The body of a `switch`, `while`, `do...while`, or `for` statement **must be a compound statement** (i.e., enclosed in `{}`), even if it contains only a single statement.
```c
// Code Snippet 1-6

// MISRA VIOLATION
while (flag)
    do_work();  // No braces

// MISRA COMPLIANT
while (flag) {
    do_work();
}
```

### 7. Compound Statements for `if` / `else`
**Category:** Required

An `if` construct must be followed by a compound statement. The `else` keyword must be followed by either a compound statement or another `if` statement. This prevents the classic **dangling else** and indentation bugs.
```c
// Code Snippet 1-7

// MISRA VIOLATION
if (condition1)
    x = 1;
else
    x = 8;
    y = 2;  // Looks like it's in the else — it is NOT

// MISRA COMPLIANT
if (condition1) {
    x = 1;
} else if (condition2) {
    if (condition3) {
        x = 23;
    } else {
        x = 2;
    }
} else {
    x = 8;
    y = 2;
}
```

### 8. All `if...else if` Chains Must End with `else`
**Category:** Required

Every `if...else if` chain must be terminated with a final `else` clause. This ensures all possible conditions are explicitly handled and makes the intent of the code clear.
```c
// Code Snippet 1-8

// MISRA VIOLATION
if (mode == MODE_A) {
    handle_a();
} else if (mode == MODE_B) {
    handle_b();
}
// What happens if mode is neither A nor B? Undefined behavior risk.

// MISRA COMPLIANT
if (mode == MODE_A) {
    handle_a();
} else if (mode == MODE_B) {
    handle_b();
} else {
    /* No action required for other modes */
}
```

> **Tip:** The `else` block can be empty with a comment explaining why, but it must be present.

### 9. No `#undef`
**Category:** Required

`#undef` removes a macro definition, which can cause subtle and hard-to-trace bugs when code relies on a macro being consistently defined throughout a translation unit. Avoid redefining or undefining macros; define them once in a header and leave them.
```c
// Code Snippet 1-9

// MISRA VIOLATION
#define BUFFER_SIZE 256
/* ... */
#undef BUFFER_SIZE
#define BUFFER_SIZE 512  // Redefinition after undef

// MISRA COMPLIANT
// Define once, never undef. Use a single authoritative value.
#define BUFFER_SIZE 256
```

### 10. Prefer Functions Over Function-Like Macros
**Category:** Advisory

Function-like macros are error-prone: they lack type checking, have unexpected behavior with side effects in arguments, and are harder to debug. Use `inline` functions or regular functions instead wherever possible.
```c
// Code Snippet 1-10

// NOT PREFERRED — function-like macro
#define SQUARE(x)  ((x) * (x))

int result = SQUARE(i++);  // i gets incremented twice — undefined behavior

// PREFERRED — inline function with type safety
static inline int square(int x) {
    return x * x;
}

int result = square(i++);  // i incremented once, as expected
```

### 11. No Dynamic Memory Allocation
**Category:** Required

`malloc`, `calloc`, `realloc`, and `free` are **strictly banned** in MISRA C. Dynamic memory allocation introduces:
- **Heap fragmentation** over time
- **Non-deterministic allocation failures** at runtime
- **Undefined behavior** on double-free or use-after-free

In safety-critical systems, all memory must be **statically allocated** so that worst-case memory usage is known at compile time.
```c
// Code Snippet 1-11

// MISRA VIOLATION
uint8_t *buf = (uint8_t *)malloc(BUFFER_SIZE);
if (buf != NULL) {
    process(buf);
    free(buf);
}

// MISRA COMPLIANT — static allocation
static uint8_t buf[BUFFER_SIZE];
process(buf);
```

## Quick Reference Table

|**Rule**|**Category**|**Summary**|
|---|---|---|
|No unreachable code|Required|Code after exit points is forbidden|
|No `goto`|Required|No arbitrary jumps|
|No `continue`|Required|No hidden loop iteration skips|
|One `break` per loop|Required|At most one loop exit point|
|Single function exit|Required|One `return` at end of function|

# Advanced Function Concepts: Reentrancy

## Reentrant vs Non-Reentrant Functions
A **reentrant function** is a function that can be safely interrupted at any point during its execution, and then resumed later without any loss of data or corruption of state. It can be called by multiple tasks or contexts concurrently without issue.

A **non-reentrant function** is one that cannot be safely interrupted and resumed; doing so risks corrupting the data it operates on.

## What is an Entrant?
Before distinguishing the two function types, it helps to understand the concept of an **entrant** (anything that can interrupt or interleave execution of running code).

### Interrupts
An **interrupt** is a signal sent to the processor when a specific hardware or software event is triggered. When an interrupt occurs:
1. The currently executing function is **paused**
2. The processor **saves its context** (registers, stack pointer, program counter)
3. An **ISR** (Interrupt Service Routine) executes
4. Once the ISR returns, the processor **restores the saved context** and resumes the paused function

This mechanism is what makes reentrancy a concern; if the ISR touches the same data that the paused function was using, the data can be left in an inconsistent state.

## The Two Function Types

| **Type**          | **Can Be Interrupted and Resumed?** | **Typical Characteristic**                           |
| ----------------- | ----------------------------------- | ---------------------------------------------------- |
| **Reentrant**     | Yes                                 | Uses only local variables and encapsulated resources |
| **Non-Reentrant** | No                                  | Uses shared or global variables / shared resources   |

## Non-Reentrant Functions
A function is non-reentrant if it relies on **shared state**, global variables, static local variables, hardware registers, or any resource that is not exclusive to the current call.

### Unsafe `swap` Using a Global Temporary
```c
// Code Snippet 1-12

int temp;  // Shared global variable

void swap(int *ptr1, int *ptr2) {
    temp = *ptr1;   // (1) save first value into global
    *ptr1 = *ptr2;  // (2) overwrite first with second
    *ptr2 = temp;   // (3) write saved value into second
}
```

This looks correct in isolation. But consider what happens if an interrupt fires between steps (1) and (3):
```c
// Code Snippet 1-13

ISR(void) {
    temp++;  // Modifies the same global temp
}
```

Execution timeline:

```bash
# Bash Snippet 1-1

swap() runs:  temp = *ptr1;       // temp = 40
--- INTERRUPT FIRES ---

ISR() runs:   temp++;              // temp = 41  (corrupted)
--- ISR RETURNS ---

swap() resumes: *ptr2 = temp;     // *ptr2 = 41, not 40 — wrong result
```

The swap produces an incorrect result because the ISR corrupted the shared `temp` variable between the two steps that depended on it. This is a classic **reentrancy bug**.

## Reentrant Functions
A function is reentrant if it uses only **local variables** (stack-allocated) and does not touch any shared or global state. Because local variables live on the stack, each call — including a re-entry from an ISR — gets its own independent copy.

### Safe `swap` Using a Local Temporary
```c
// Code Snippet 1-14

void swap(int *ptr1, int *ptr2) {
    int temp;       // Local variable — lives on the stack, private to this call
    temp = *ptr1;
    *ptr1 = *ptr2;
    *ptr2 = temp;
}
```

Now if an interrupt fires mid-execution, the ISR has no access to this `temp` — it is private to the current stack frame. When the ISR returns and `swap` resumes, `temp` holds exactly the value it had before the interrupt, and the function completes correctly.

## Reentrancy in RTOS
Reentrancy becomes especially critical in **Real-Time Operating Systems (RTOS)**, where multiple tasks run concurrently and can preempt each other based on priority.
### How RTOS Scheduling Creates the Problem
In an RTOS, each task has a priority level. The scheduler runs the highest-priority ready task at any given moment. Consider:
- **Task 1** -> high priority
- **Task 2** -> low priority, currently executing

If Task 1 becomes ready while Task 2 is mid-execution, the scheduler **preempts Task 2** — pausing it wherever it is in its code — and hands control to Task 1. Once Task 1 finishes, Task 2 resumes from where it was cut off.

If both tasks call the same non-reentrant function (one that relies on a shared global), they will interfere with each other's data, producing incorrect and often non-deterministic results.
### Why These Bugs Are Difficult to Find
Reentrancy bugs in RTOS environments are particularly hard to detect because:

- They are **timing-dependent** — the bug only appears when a preemption happens at exactly the wrong moment
- They are **non-deterministic** — the same code can run correctly hundreds of times and then fail unexpectedly
- They often surface only in **assembly-level analysis**, where the exact instruction boundary at which preemption occurs becomes visible
- Reproducing them in a test environment requires precise control over task scheduling, which is rarely achievable without specialized tooling

This makes reentrancy one of the more demanding correctness properties to verify, requiring a high level of knowledge in both systems programming and concurrent software testing.

**Quick Rule to Check:** if a function's correctness depends on state that exists outside its own stack frame, it is non-reentrant.

# **Synchronous vs Asynchronous Functions in C and Embedded C**

This section is the final part of the **C and Embedded C Guide**, covering critical concepts for professional embedded systems development.
## SYNC vs ASYNC Functions
In Embedded programming, the way functions execute and return control significantly impacts system performance, power consumption, and real-time behavior.

- **Synchronous functions** complete their work before returning.
- **Asynchronous functions** start an operation and return immediately, notifying later.

This guide also explains **polling**, **busy-waiting**, **direct execution**, and **callbacks** with detailed code walkthroughs.
### 1. Synchronous Functions
A **synchronous function** initiates the requested functionality and **returns with the result** only after the task is fully completed.

#### Characteristics:
- Blocking: The caller waits until the function finishes.
- Simple and deterministic execution flow.
- Often uses **polling** or **busy-waiting**.
#### What is Polling and Busy-Waiting?
- **Polling**: Continuously checking the status of a hardware peripheral in a loop.
- **Busy-Waiting**: A form of polling where the CPU is kept busy doing nothing useful while waiting (wastes CPU cycles and power).
#### Code Example
```c
// Code Snippet 1-15

uint8 UART_receiveByte(void)
{
    /* Busy-wait until the UART receives a byte */
    while (UART_IsDataAvailable() == 0)   // Polling loop
    {
        // CPU is stuck here doing nothing (busy-waiting)
        // This wastes power and prevents the CPU from doing other tasks
    }
    
    return UART_ReadReceivedByte();   // Read the actual data
}

void getReceivedData(uint8 *data)
{
    *data = UART_receiveByte();   // This line blocks until a byte arrives
}
```

**Step-by-step explanation**:
1. `UART_receiveByte()` is called.
2. The function enters a `while` loop and keeps checking the UART status register (polling).
3. As long as no data is available, the CPU spins in this loop → **busy-waiting**.
4. Once a byte arrives, the loop exits.
5. The function reads the byte and returns it.
6. Only then does the caller (`getReceivedData`) regain control.

**Pros**: Easy to write and understand.
**Cons**: Inefficient for long waits (e.g., waiting for human input or slow communication).
**When to use**: Short operations where the wait time is very brief and predictable.
### 2. Direct Execution (Synchronous)
Some synchronous functions perform the entire operation **directly** without waiting for hardware. This is the simplest form of synchronous behavior.
#### Example:
```c
// Code Snippet 1-16

void GPIO_TogglePin(uint8 pin)
{
    // Direct execution - no waiting
    GPIO_WritePin(pin, !GPIO_ReadPin(pin));
    
    // Function returns immediately after toggling
}
```
**Explanation**: The function executes the toggle operation in a few CPU instructions and returns right away. No polling or waiting involved.
### 3. Asynchronous Functions
An **asynchronous function** initiates the operation and **returns immediately** without waiting for completion.
#### Characteristics:
- Non-blocking.
- Completion is signaled later via **callbacks**, **interrupts**, or events.
- Allows the CPU to perform other tasks or enter low-power mode.
#### Code Example with Detailed Explanation
```c
// Code Snippet 1-17

void Timer_Start(uint32 milliseconds, void (*callback)(void))
{
    /* 1. Configure the timer hardware */
    Timer_SetPeriod(milliseconds);
    Timer_RegisterCallback(callback);   // Save the function pointer
    
    /* 2. Start the timer */
    Timer_Enable();
    
    /* 3. Return immediately - do NOT wait for timer to expire */
}

void MyTaskCallback(void)
{
    /* This function runs when timer expires (called from ISR) */
    LED_Toggle();
    // Perform any other completion work
}
```

**Usage**:
```c
// Code Snippet 1-18

int main(void)
{
    Timer_Start(1000, MyTaskCallback);   // Starts timer and returns immediately
    // CPU is free to do other work here
    while(1)
    {
        // Main loop continues running
    }
}
```

**Step-by-step explanation**:
1. `Timer_Start()` configures the hardware timer and registers the callback (saves the function pointer).
2. It starts the timer and **returns control immediately** to the caller.
3. Later, when the timer hardware counts down to zero, it triggers an interrupt.
4. The Interrupt Service Routine (ISR) calls the registered callback function (`MyTaskCallback`).
5. The callback executes the desired action (e.g., toggle LED).

### 4. Callback Functions
A **callback** is a normal function passed as a pointer so that lower layers can call upper layers later.
```c
// Code Snippet 1-19

typedef void (*Callback_t)(void);   // Callback type

static Callback_t timerCallback = NULL;

void Timer_RegisterCallback(Callback_t cb)
{
    timerCallback = cb;   // Store the pointer
}

void Timer_IRQHandler(void)   // Called by hardware interrupt
{
    if (timerCallback != NULL)
        timerCallback();   // Execute the upper layer function
}
```

**Key Point**: Callbacks reverse the normal call direction — from lower layers (drivers) to upper layers (application).

## 5. Layered Architecture
```bash
# Bash Snippet 1-2
Application Layer
        ↓ (Direct API calls)
Hardware Abstraction Layer (HAL / HWAL)
        ↓ (Direct calls)          ↑ (Callbacks / ISRs)
Microcontroller Abstraction Layer (MCAL) - Drivers (UART, Timer, GPIO...)
```

- **Downward**: Normal function calls (APIs).
- **Upward**: Callbacks or interrupts.
### Comparison Table

|**Aspect**|**Synchronous (Polling/Busy-Wait)**|**Asynchronous (Callback/Interrupt)**|
|---|---|---|
|Blocking|Yes|No|
|CPU Efficiency|Poor (busy-waiting)|Excellent|
|Responsiveness|Low|High|
|Code Complexity|Low|Medium|
|Power Consumption|High|Low|
|Best For|Short & simple operations|Peripherals, timing, communication|

### Best Practices
- Use **synchronous** functions during initialization and for quick operations.
- Prefer **asynchronous** with interrupts/callbacks for most peripheral operations in real applications.
- Avoid long busy-wait loops in production code.
- Keep callbacks short and fast — defer heavy work to the main loop if possible.
- Always document whether a function is blocking or non-blocking.
