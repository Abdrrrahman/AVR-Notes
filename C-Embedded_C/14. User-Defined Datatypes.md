- Structures

# Structures
Struct (Structures) is similar to arrays, but arrays are collections of the same datatype, while structures are of different datatypes. Each member it holds can be defined inside it with a specific datatype you choose. 

Array is a derived datatype, not a user-defined datatype, while Struct is a user-defined datatype. Structs are defined with the keyword `struct`, and then comes the name of the struct which we call `tagname`. It needs a `;` after defining it.
We type the members we want inside its scope. This is a code snippet for a struct declaration (just defining the user-defined datatype):
```c 
// Code Snippet 1-1
 
struct car { 
	int id; 
	double cost; 
};
```
Normally, the struct definition is put in a header file that you include, especially if you have many files that use this struct.

The `id` and `cost` are called members or attributes. This program doesn't allocate any memory in RAM or Flash (you didn't make any objects); it only specifies the blueprint for any car you make after.
```c
// Code Snippet 1-2

int main(void) {
    struct car myCar;
}
```

Here, you have now allocated an object of the `car` user-defined datatype.

What about the lifetime, memory segments, and scope of each member inside `myCar`?
They are typically as usual with no differences: the scope is inside `main`, and the memory segment is on the stack frame because it is a local, non-static variable, and so on.

## Initialization During Declaration
In the above code snippet 1-2, we successfully declared an object from the struct. How do we initialize it with values during declaration?
```c
// Code Snippet 1-3

int main(void) {
    struct car myCar = {5, 15.0};
}
```

It is like an array: we just open curly braces and type in the values we want to assign to each member respectively.

Can we declare 2 struct variables in the same line?
```c
// Code Snippet 1-4

int main(void) {
    struct car myCar, hisCar;
}
```

## Accessing Members
You can access each member of the struct you defined using the dot operator `.`

The dot operator and arrow operator have the highest precedence among all operators.
```c
// Code Snippet 1-5

int main(void) {
    struct car myCar = {5, 15.0};
    myCar.id = 50;
    myCar.cost = 100.1 + myCar.id;
}
```

You can create more than one struct datatype in the same program:
```c
// Code Snippet 1-6

struct car {
    int id;
    double cost;
};

struct client {
    int id;
    double cash;
};

int main(void) {
    struct car myCar  = {1, 15.0};
    struct client myClient = {5, 20.0};
}
```

You can create the object during the declaration, and it will be a global variable:
```c
// Code Snippet 1-7

struct car {
    int id;
    double cost;
} myCar;
```

You can initialize it as well:
```c
// Code Snippet 1-8

struct car {
    int id;
    double cost;
} myCar = {1, 15.0};
```

You can declare the structure inside the `main` function, but the scope of the variables you create will be inside `main` only:
```c
// Code Snippet 1-9

int main(void) {
    struct car {
        int id;
        double cost;
    };
}
```

You can delete the `tagname`, but you must declare some objects during declaration:
```c
// Code Snippet 1-10

struct {
    int id;
    double cost;
} x, y;

int main(void) {
    /* You can't create objects from the struct above, as there is no tagname 
       for it */
}
```

Structs Can Hold Constants

## Initializing a Single Member

You can initialize a member explicitly using designated initializers and leave other members uninitialized:
```c

```

```c
struct car {

}
```

## You Can Have Strings Inside the Structs
```c
// Code Snippet 1-15

struct car {
    int id;
    char company[50];
    char model[50];
};

int main(void) {
    struct car myCar = {.company = {'N', 'i', 's', 's', 'a', 'n'}, .model = "Sunny"};
}
```

If you want to change the model after initializing, you need to use `strcpy`:
```c
// Code Snippet 1-16

struct car {
    int id;
    char company[50];
    char model[50];
};

int main(void) {
    struct car myCar = {.company = {'N', 'i', 's', 's', 'a', 'n'}, .model = "Sunny"};
    myCar.company = "Toyota"; // This is a compilation error
}
```

`myCar.company` is a char array attribute and cannot be assigned a string literal (`const char *`). You cannot assign a pointer to an array with `=`.

The biggest difference between a char pointer and a char array is that arrays own the memory they point to, while pointers don't. Pointers can be reassigned, while arrays cannot.

You might think that the safest solution is to replace the char array with a pointer to char (`char *`):
```c
// Code Snippet 1-17

struct car {
    int id;
    char *company;
    char *model;
};

int main(void) {
    struct car myCar = {.company = "Nissan", .model = "Sunny"};
    myCar.company = "Toyota"; // This will work
}
```

This solution is good and won't throw a compile error, although it is not entirely safe. This solution has a problem with copying: if we defined two structs and tried to assign one of them to the other, it would perform a shallow copy.
```c
// Code Snippet 1-18

struct car {
    int id;
    char *company;
    char *model;
};

int main(void) {
    struct car myCar = {.company = "Nissan", .model = "Sunny"};
    struct car hisCar = {.company = "Toyota", .model = "Corolla"};
    
    hisCar.company = myCar.company;
    // This is shallow copying
}
```

Shallow copying means that the two objects will point to the same string, which is a failure in struct safety.

The best solution here is to use `strcpy` for reassigning:
```c
// Code Snippet 1-19

#include <string.h>

struct car {
    int id;
    char company[50];
    char model[50];
};

int main(void) {
    struct car myCar = {.company = "Nissan", .model = "Sunny"};
    
    strcpy(myCar.company, "Toyota");
}
```

## Struct Copy
You can copy an object to another if and only if they are of the same struct type:
```c
// Code Snippet 1-20

struct car {
    int id;
    char company[50];
    char model[50];
};

struct bus {
    int id;
    char company[50];
    char model[50];
};

int main(void) {
    struct car myCar = {.company = "Nissan", .model = "Sunny"};
    struct car hisCar = {.company = "Toyota", .model = "Corolla"};
    struct bus myBus = {.company = "Mercedes", .model = "Bus01"};
    
    myCar = hisCar; // This works
    hisCar = myBus; // This is a compilation error
}
```

You cannot assign a bus to a car even though they have exactly the same members.

You cannot compare structs, add them, or subtract them:
```c
// Code Snippet 1-21

struct car {
    int id;
    char company[50];
    char model[50];
};

int main(void) {
    struct car myCar = {.company = "Nissan", .model = "Sunny"};
    struct car hisCar = {.company = "Toyota", .model = "Corolla"};

    if (myCar == hisCar) // Compilation error 
    {
        myCar++; // Compilation error
        struct car ourCar = hisCar + myCar; // Compilation error
    }
}
```

## Passing Structs to Functions

You can pass structs to functions using pass-by-value and pass-by-reference, but preferably use pass-by-reference to save execution time and memory.
```c
// Code Snippet 1-22

struct car {
    int id;
    double cost;
};

void addToCar(struct car otherCar);

int main(void) {
    struct car myCar = {15, 109.3};
    
    addToCar(myCar);
}

void addToCar(struct car otherCar) {
    otherCar.id += 100;
}
```
This is a classic call-by-value as the function `addToCar` is copying the object `myCar` into another object named `otherCar`.
This is useful for the array pass-by-value trick.

In C, arrays cannot be passed by value; they are always passed by reference because the name of the array decays into a pointer to the array.

One solution is to `memcpy()` the array inside the function to another array, but a better solution is to use a wrapper struct and pass this wrapper struct by value.
```c
// Code Snippet 1-23

#include <stdio.h>

typedef struct {
    int data[5];
} Array;

void modify(Array a) {
    a.data[0] = 999;  // modifies the LOCAL copy only
}

int main(void) {
    Array a = {{1, 2, 3, 4, 5}};
    modify(a);
    printf("%d\n", a.data[0]);  // still prints 1
}
```

You can return a struct also:
```c
// Code Snippet 1-24

struct car {
    int id;
    double cost;
};

struct car addToCar(struct car otherCar);

int main(void) {
    struct car myCar = {15, 109.3};
    
    myCar = addToCar(myCar);
}

struct car addToCar(struct car otherCar) {
    otherCar.id += 100;
    return otherCar;
}
```

## Pointer to a Structure
You can access a structure object using a pointer to its object:
```c
// Code Snippet 1-25

struct car {
    int id;
    double cost;
};

int main(void) {
    struct car myCar = {10, 20.4};
    struct car *p_car = &myCar;
    *p_car.id = 100; // Compilation error: precedence problem
    // It will execute the dot operator first
    
    (*p_car).id = 100; // Works
    p_car->id = 100;   // Works
}
```

You can pass structures by pointers to avoid copying and allow functions to edit them:
```c
// Code Snippet 1-26

struct car {
    int id;
    double cost;
};

void addToCar(struct car *p_car);

int main(void) {
    struct car myCar = {15, 109.3};
    
    addToCar(&myCar);
}

void addToCar(struct car *p_car) {
    p_car->id += 100;
}
```
It is better in most use cases to pass structures by reference (pointer) to save both execution time and memory.

# `typedef` Keyword

The type definition keyword gives you the ability to replace data types with new names you define.
```c
// Code Snippet 1-31

#include <stdio.h>

typedef signed char Byte;

int main(void) {
    Byte x = 255;
    signed char y;
    // y is of the same type of x
    printf("%i\n", x); // prints -1 as it is a byte full of 1 bits
}
```

By convention, the new name of the data type is uppercase as a good practice. The new names are not arbitrarily chosen; there are standard names that we will use and they must be placed inside a header file and included in the source file.
```c
// Code Snippet 1-32

// typedef.h
#pragma once
typedef unsigned char uint8_t;
typedef signed char sint8_t;
typedef unsigned short uint16_t;
typedef signed short sint16_t;
typedef unsigned long uint32_t;
typedef signed long sint32_t;
typedef unsigned long long uint64_t;
typedef signed long long sint64_t;
typedef float float32_t;
typedef double float64_t;
```

```c
// Code Snippet 1-33

// main.c
#include <stdio.h>
#include "typedef.h"

int main(void) {
    uint8_t x = 255;
    sint8_t y = 255; 
    printf("x=%i\ny=%i\n", x, y);
    // x=255
    // y=-1
    return 0;
}
```
You cannot `typedef` `auto`; this is a compilation error.

You can manipulate it with arrays:
```c
// Code Snippet 1-34

#include <stdio.h>

typedef int arr[10];

int main(void) {
    arr myArr;
    myArr[1] = 8;
    printf("%i\n", myArr[1]);
    // prints 8
    return 0;
}
```
Here, `arr` has become a type alias to represent an array of 10 integers.

## `typedef` vs `#define`
`#define` is a text replacement that occurs during the preprocessing stage, while `typedef` is interpreted during compilation time. To show the difference, let's examine this code:
```c
// Code Snippet 1-38

#include <stdio.h>

typedef int *ptr1_t;
#define ptr2_t int *

int main(void) {
    int x = 49, y = 19;
    ptr1_t p1x = &x, p1y = &y;
    ptr2_t p2x = &x, p2y = &y; // Compilation Error on p2y
    
    printf("%i\n%i\n", *p1y, *p2y);
    return 0;
}
```
The first declaration compiles into normal code where both `p1x` and `p1y` are pointers to `int`. The second line compiles such that `p2x` is a pointer to `int`, but `p2y` is a normal `int`. Trying to dereference it will result in a compilation error.
```c
// Code Snippet 1-39

ptr2_t p2x = &x, p2y = &y;
// This transforms into:
int *p2x = &x, p2y = &y; // matches int *p2x = &x; int p2y = &y;
```
So, `typedef` is different from `#define`; they are not the same.

# Structure Padding
If we have a processor with a 32-bit data bus, its natural alignment boundary is 4 bytes, meaning it can read/write 4 bytes at the exact same time. This is efficient because the processor can pull multiple smaller variables in a single instruction loop cycle.

We can alter alignment behaviors using directives like `#pragma pack(1)`. If a structure contains mixed-type members, padding bytes are inserted by default.

The standard layout rules dictate that:
- The starting address of a structure object is typically aligned to a multiple of its largest member's size.
- Each member is placed at an offset address divisible by its individual alignment size.
- The structure's total size is rounded up to a multiple of its maximum member alignment requirement.
```c
// Code Snippet 1-40

typedef struct {
    char x;
    short y;
    char z;
} Person; 
```

Without padding, the structure would look like this in raw memory:

```bash
# Bash Snippet 1-1

100 -----------
                  char x
101 -----------
                
102 -----------   short y
                  
103 -----------
                  char z
104 -----------
```

If the processor reads in strict 2-byte aligned blocks:
1. First access pulls addresses `100` and `101`, grabbing `char x` and the first half of `short y`.
2. Second access pulls addresses `102` and `103`, grabbing the second half of `short y` and `char z`.    

This mismatched alignment forces the processor/compiler to issue masking and shifting instructions behind the scenes to reconstruct values. To prevent performance loss, the compiler pads out memory layouts smoothly at the expense of footprint:
```bash
# Bash Snippet 1-2

100 -----------
                  char x
101 -----------   [Padding Byte]
                
102 -----------   
                  short y
103 -----------   
                  
104 -----------
                  char z
105 -----------   [Padding Byte]
```

It is highly recommended to rearrange your structure definitions manually by size to minimize padding footprint without altering hardware compliance configurations.

Ascending layout:
```c
// Code Snippet 1-41

typedef struct {
    char x;
    char z;
    short y;
} Person; 
```

Descending layout:
```c
// Code Snippet 1-42

typedef struct {
    short y;
    char x;
    char z;
} Person; 
```

In these scenarios, grouping matching bounds together eliminates intermediate alignment holes cleanly.

### `#pragma pack(1)`
**`#pragma pack(1)`** is a compiler-specific directive that forces structure packing down to a single byte layout level.
#### What It Does
By default, compilers add padding bytes to align elements smoothly. `#pragma pack(1)` overrides this behavior by grouping structures as tightly as possible, erasing padding spaces.
#### Example
```c
// Code Snippet 1-43

struct Normal {
    char  a;   // 1 byte
    int   b;   // 4 bytes
    short c;   // 2 bytes
}; 
// Size is usually 12 bytes due to default padding gaps

#pragma pack(1)
struct Packed {
    char  a;
    int   b;
    short c;
};
#pragma pack(pop)   
// Size is exactly 7 bytes (1 + 4 + 2)
```

#### Common Usage Patterns
```c
// Code Snippet 1-44

#pragma pack(push, 1)
struct MyStruct { 
    char element;
    int data;
};
#pragma pack(pop)     
```

#### Why Use It?
- **Binary Compatibility**: To match precise configurations expected by network byte streams, binary file maps, or external peripheral registers.
- **Memory Constraints**: Minimizing footprints safely inside ultra-lean systems.

#### Important Downsides
1. **Performance Penalties**: Accessing unaligned fields requires multiple bus cycles on specific processors (like ARM architectures).
2. **Code Bloat**: The compiler might generate extra instructions to parse fragmented fields safely.
3. **Portability Shifts**: Packing directives can behave differently across distinct toolchains.

### `#pragma pack(push, 1)` and `#pragma pack(pop)`
These variations ensure that changes made locally inside source structures do not leak unintentionally into surrounding header inclusions.
- **`push`**: Saves the active global configuration settings state safely onto an internal compilation tracking stack.
- **`pop`**: Restores the immediately preceding configuration setting smoothly.

#### Full Recommended Pattern
```c
// Code Snippet 1-45

#include <stdint.h>

#pragma pack(push, 1)

struct MyBinaryStruct {
    uint8_t  type;
    uint32_t length;
    uint16_t flags;
} __attribute__((aligned(1))); // Explicit attribute enforcement on GCC/Clang systems

#pragma pack(pop)
```
The attribute expression `__attribute__((aligned(1)))` enforces that the overall instance starting address requirements can fall down onto arbitrary odd byte limits cleanly without throwing configuration warnings.

|**Feature**|**Purpose**|**Main Compiler Support**|
|---|---|---|
|`#pragma pack(1)`|Erases structure padding spaces|All major systems (MSVC, GCC, Clang)|
|`__attribute__((aligned(1)))`|Forces minimum outermost structure base address limits|GCC / Clang|

#### Other Useful Variations

| **Directive**           | **Meaning**                                                          |
| ----------------------- | -------------------------------------------------------------------- |
| `#pragma pack(push, 1)` | Saves current layout context and limits padding gaps to 1 byte.      |
| `#pragma pack(pop)`     | Restores previous global state parameters directly.                  |
| `#pragma pack(push, 4)` | Saves current environment states and forces 4-byte boundaries.       |
| `#pragma pack(1)`       | Modifies layout globally without preserving context (High Bug Risk). |

# Unions
Unions are similar to structures, they are user-defined datatype also.
```c
// Code Snippet 1-46

#include <stdio.h>

union car {
    int id;
    short chassetNum;
    double cost;
} car1;

int main(void) {
    union car car2 = {.id = 14, .chassetNum = 933, .cost = 1400.0};
    car1 = car2;
    car1.id++;
    printf("The size of the union is: %zu", sizeof(union car));
    // prints out 8
}
```

The biggest difference between unions and structures is that unions literally union the memory between its members. In the above example, we have `int`, `short`, and `double` inside the union.

The union allocates shared memory between them, meaning that its size will be that of the largest datatype in it, which is `double`, so it will be 8 bytes long. If you try to use `int`, it will use the first 4 bytes; if you try to use `short`, it will use the first 2 bytes; if you try to use `double`, it will use all 8 bytes.

It uses overlapped memory, which presents an obvious overwriting problem. So, why does such a thing exist?
It may be useful for some cases. If I have a variable in embedded systems that I need to use at one time as an `int` and then another time as a `float`, I can make a union of the 2 datatypes.
It provides an efficient way of using the same memory address for multiple purposes.

# Enums
Enumerations are a user-defined data type that gives you the ability to represent strings using integers.
```c
// Code Snippet 1-47

enum week {
    Saturday = 10,
    Sunday = 1,
    Monday = 2,
    Tuesday = 6,
    Wednesday = 4,
    Thursday = 5,
    Friday = 9
};

int main(void) {
    enum week day;
    day = Saturday; // 10
    day = 10;       // Saturday 
}
```

The variable named `day` is of size `int` (4 bytes). If you have a member inside the enum with an assigned value greater than $2^{31}-1$ (or outside the range of `int`), then the underlying type of the enum can be promoted into something bigger than `int`, such as `long long`.

Enums are considered safer than using strings. Since strings can have typos, enums are always used when we have multiple options and we choose from them.

You don't need to specify a value for each day; you can just type the members and they will be numbered ascendingly from zero:
```c
// Code Snippet 1-48

enum week {
    Saturday,
    Sunday,
    Monday,
    Tuesday,
    Wednesday,
    Thursday,
    Friday
};

int main(void) {
    enum week day;
    day = Saturday; // 0
    day = 0;        // Saturday 
}
```

## Enums vs `#define`
`#define` is not limited to assigning strings with integers; it can be used for a wide headcount of options, like macro-like functions.

In the case of assigning strings with integers, what is the difference between them?
`#define` replacement is done during the Preprocessing stage, while enum replacement is processed during the Compilation stage.
This is not the biggest difference, as both of them are resolved during Compile Time.
What else?
Enums cannot hold fractions or floats, while `#define` can.
The biggest difference is readability.
## Readability
In terms of readability, enumerations make better constants than macros because related values are grouped together. In addition, `enum` defines a new type, so readers of your program will have an easier time figuring out what can be passed to the corresponding parameter.

Using `#define` macros:
```c
// Code Snippet 1-49

#define UNKNOWN  0
#define SUNDAY   1
#define MONDAY   2
#define TUESDAY  3
...
#define SATURDAY 7
```

Using an `enum`:
```c
// Code Snippet 1-50

enum Weekday {
    UNKNOWN,
    SUNDAY,
    MONDAY,
    TUESDAY,
    ...
    SATURDAY
};
```

It is much easier to read code like this:
```c
// Code Snippet 1-51

void calendar_set_weekday(enum Weekday wd);
```

than this:
```c
// Code Snippet 1-52
void calendar_set_weekday(int wd);
```

Using `Weekday` as the parameter type makes the intent immediately clear to anyone reading the function signature; they know exactly what values are valid to pass.

You can have 2 members in the same enum having the same number assigned to them. This is useful for backward compatibility:
```c
// Code Snippet 1-53

enum Status {
    OK = 0,
    SUCCESS = 0,  // Alias — both are 0, totally valid
    FAILURE = 1
};
```

If you have an enum with some initialized members and some other uninitialized, the uninitialized takes the value of its previous member $+ 1$:
```c
// Code Snippet 1-54

enum week {
    Saturday = 50,
    Sunday,
    Monday,
    Tuesday = 2,
    Wednesday = 1,
    Thursday = 0,
    Friday
};
```
Here, `Sunday` is 51, `Monday` is 52, and `Friday` is 1.

In C, enums are not truly scoped types. All enum members are placed directly into the **enclosing scope** (usually global or file scope) as plain integer constants.

```c
// Code Snippet 1-55

typedef enum { RED, GREEN, BLUE } Color;
typedef enum { RED, YELLOW } Fruit;  // error: redefinition of 'RED'
```
Both `RED`s end up as the same-scope integer constant. The compiler sees this as redefining the same name twice, which is illegal.

This is fundamentally different from C++ where enum classes are used to scope members:
```cpp
// Code Snippet 1-56

enum class Color { RED, GREEN, BLUE };
enum class Fruit { RED, YELLOW };      // Works fine
// Color::RED vs Fruit::RED
```

**The C workaround** is to manually prefix your enum members:
```c
// Code Snippet 1-57

typedef enum { COLOR_RED, COLOR_GREEN, COLOR_BLUE } Color;
typedef enum { FRUIT_RED, FRUIT_YELLOW }             Fruit;
```
This is why there are prefixes like `GL_`, `SDL_`, `HTTP_` everywhere in C libraries.

# Bit Fields
You can make variables with a discrete number of bits (1 bit, 2 bits, etc.) using bit fields. Bit fields must be declared inside a structure.

Unlike some other computer languages, C has a built-in feature called a bit field that allows you to access individual bits tightly.

Bit fields can be useful if memory storage is limited; you can store several Boolean variables in a single byte.

A bit field must be a member of a structure or a union.
```c
// Code Snippet 1-58

struct car {
    unsigned long x:3;
    unsigned long y:6;
    unsigned long z:15;
    unsigned long w:7;
} myCar;

int main(void) {
    myCar.x = 7; // Maximum number it can hold as it is 3 bits
    return 0;
}
```
It cannot be a `float` or `double`; it can only be an integer type like `int`, `short`, `long`, or `long long`.

In this example, due to structure padding, this struct takes the alignment of the largest datatype it has. Thus, the `myCar` object allocates 4 bytes in memory, which are divided as follows:

_First Byte_:
- The first 3 bits are allocated for the variable `x`.    
- The remaining 5 bits + 1 bit from the second byte are allocated for the variable `y`.

_Second Byte_:
- The remaining 5 bits of `y` occupy the lower part of this byte.
- The rest of the bits (7 bits) + the whole third byte are allocated for the variable `z`.

_Third Byte_:
- Contains the continuation of the variable `z`.

_Fourth Byte_:
- The lower 7 bits are allocated for the variable `w`.
- The most significant bit will be padded/unused.

If we tried to add another variable of size more than 1 bit in the above example, such as `long r:2;`, a new 4 bytes would be allocated (due to alignment) and the lower 2 bits of the new allocation would belong to `r`.
```c
// Code Snippet 1-59

#include <stdio.h>

typedef struct {
    unsigned long x;
    unsigned long y;
} first;

typedef struct {
    unsigned long x:1;
    unsigned long y:1;
} second;

int main(void) {
    printf("Size of normal structure: %zu\n", sizeof(first));
    printf("Size of Bit field structure: %zu\n", sizeof(second));
    return 0;
}
```

Output:
```bash
# Bash Snippet 1-3

Size of normal structure: 8
Size of Bit field structure: 4
```

```c
// Code Snippet 1-60

#include <stdio.h>

typedef struct {
    unsigned long x:25;
    unsigned long y:1;
} first;

typedef struct {
    unsigned long x:25;
    unsigned char y:1;
} second;

int main(void) {
    printf("Size of first structure: %zu\n", sizeof(first));
    printf("Size of second structure: %zu\n", sizeof(second));
    return 0;
}
```

Output:
```bash
# Bash Snippet 1-4

Size of first structure: 4
Size of second structure: 8
```
Because the 2 bit fields of the second structure have different underlying datatypes (`unsigned long` vs `unsigned char`), the compiler aligns them according to the larger type's boundary conditions. This behavior can be highly compiler-dependent.

Some compilers allocate only 1 byte for certain combinations, while others allocate full words.
```c
// Code Snippet 1-61

typedef struct {
    unsigned long x : 5;
    unsigned long y : 6;
} first;
```

## Example 14-2
```c
// Code Snippet 1-63

#include <stdio.h>

struct person {
    unsigned char age:3;
    unsigned char number:6;
} Age;
```
To understand the binary layout clearly, we look at the exact memory allocations under different optimization/alignment scenarios:
### Scenario 1: Using `#pragma pack(1)`
```c
// Code Snippet 1-64

#include <stdio.h>

#pragma pack(push, 1)
struct person {
    unsigned char age:3;
    unsigned char number:6;
} Age = {5, 1};
#pragma pack(pop)

int main(void) {
    unsigned char *ptr = (unsigned char *)&Age;
    printf("%i\n", *(ptr));
    printf("%i\n", *(ptr + 1));
    return 0;
}
```
In this code, compiler structure packing is active, so standard alignment rules are modified.

We have an `age` variable with a size of 3 bits, and a `number` variable with a size of 6 bits. This spans a total of 9 bits, so the compiler consumes 2 bytes.

_First Byte_: `00001101`
- The first 3 bits (LSB) are allocated for the variable `age` (`5` $\rightarrow$ `101`).    
- The remaining 5 bits are filled with the lower 5 bits of the variable `number` (`1` $\rightarrow$ `00001`).

_Second Byte_: `00000000`
- The remaining 1 bit of `number` overflows into the LSB of the second byte.

When we use our pointer to look at the raw bytes, the first byte prints `13` (`00001101`), and the second byte prints `0`.

Output:
```bash
# Bash Snippet 1-6

13
0
```

### Scenario 2: Not Using `#pragma pack(1)`
```c
// Code Snippet 1-65

#include <stdio.h>

struct person {
    unsigned char age:3;
    unsigned char number:6;
} Age = {5, 1};

int main(void) {
    unsigned char *ptr = (unsigned char *)&Age;
    printf("%i\n", *(ptr));
    printf("%i\n", *(ptr + 1));
    return 0;
}
```
When packing is inactive, default compiler alignment rules kick in.

We have the same variables, but the compiler chooses not to split the `number` bitfield across a structural allocation boundary unit (byte boundary for `char`).

_First Byte_: `00000101`
- The first 3 bits are allocated for `age`.    
- The remaining 5 bits cannot fit the entire 6-bit `number` field, so they are padded out as zero.

_Second Byte_: `00000001`
- The full 6 bits for `number` are allocated at the start of the second byte.

Output:
```bash
# Bash Snippet 1-7

5
1
```
### Scenario 3: Using `#pragma pack(1)` and a Third Variable of Size 2
```c
// Code Snippet 1-66

#include <stdio.h>

#pragma pack(push, 1)
struct person {
    unsigned char age:3;
    unsigned char number:6;
    unsigned char year:2;
} Age = {5, 1, 3};
#pragma pack(pop)

int main(void) {
    unsigned char *ptr = (unsigned char *)&Age;
    printf("%i\n", *(ptr));
    printf("%i\n", *(ptr + 1));
    return 0;
}
```
With strict packing active, all 11 bits are squeezed continuously across 2 bytes.

_First Byte_: `00001101` (Prints `13`)

_Second Byte_: `00000110` (Remaining 1 bit from `number` + 2 bits from `year` $\rightarrow$ `0110` $\rightarrow$ Prints `6`)

Output:
```bash
# Bash Snippet 1-8

13
6
```

### Scenario 4: Not Using `#pragma pack(1)` and a Third Variable of Size 2
```c
// Code Snippet 1-67

#include <stdio.h>

struct person {
    unsigned char age:3;
    unsigned char number:6;
    unsigned char year:2;
} Age = {5, 1, 3};

int main(void) {
    unsigned char *ptr = (unsigned char *)&Age;
    printf("%i\n", *(ptr));
    printf("%i\n", *(ptr + 1));
    return 0;
}
```
Without packing, `age` is in byte 1. `number` cannot fit in byte 1, so it moves to byte 2. `year` (2 bits) fits cleanly into the remaining unallocated upper bits of byte 2.

_First Byte_: `00000101` (Prints `5`)

_Second Byte_: `11000001` (`number` in lower 6 bits + `year` in upper 2 bits $\rightarrow$ `11000001` $\rightarrow$ Prints `193`)

Output:
```bash
# Bash Snippet 1-9

5
193
```

### Scenario 5: Using `#pragma pack(1)` and a Third Variable of Size 3
```c
// Code Snippet 1-68

#include <stdio.h>

#pragma pack(push, 1)
struct person {
    unsigned char age:3;
    unsigned char number:6;
    unsigned char year:3;
} Age = {5, 1, 3};
#pragma pack(pop)

    printf("%i\n", *(ptr));

    return 0;
}
```

Output:
```bash
# Bash Snippet 1-10

13

```

### Scenario 6: Not Using `#pragma pack(1)` and a Third Variable of Size 3
```c
// Code Snippet 1-69

#include <stdio.h>

struct person {
    unsigned char age:3;
    unsigned char number:6;
    unsigned char year:3;
} Age = {5, 1, 3};

int main(void) {
    unsigned char *ptr = (unsigned char *)&Age;
    printf("%i\n", *(ptr));
    printf("%i\n", *(ptr + 1));
    printf("%i\n", *(ptr + 2));
    return 0;
}
```
Here, the compiler sees that the remaining 2 bits of the second byte are not enough to hold the 3-bit `year` field, so it leaves them as padding and forces `year` into a completely new third byte.

Output:
```bash
# Bash Snippet 1-11

5
1
3
```
