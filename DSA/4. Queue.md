# Queue Overview
A **Queue** is a linear data structure that stores elements in a sequential manner. Unlike a stack, where insertion and deletion occur at the same end of the structure, a queue performs insertion and deletion at **opposite ends**.

Insertion always occurs at the **rear** of the queue, while deletion always occurs from the **front**.

Because of this behavior, a queue follows the **FIFO (First In, First Out)** principle. This means that the first element inserted into the queue will also be the first element removed and the last element inserted into the queue will also be the last element removed.

Queues are commonly used whenever resources or tasks must be processed in the order they arrive. Examples include printer job scheduling, customer service systems, operating system process scheduling, and breadth-first search (BFS) in graphs.

## FIFO Principle
Suppose we enqueue the following elements:
```c
// Code Snippet 1-1

Enqueue(10)
Enqueue(20)
Enqueue(30)
Enqueue(40)
```

The queue becomes
```bash
# Bash Snippet 1-1

Front                                  Rear
  │                                      │
  ▼                                      ▼
+----+    +----+    +----+    +----+
| 10 | -> | 20 | -> | 30 | -> | 40 |
+----+    +----+    +----+    +----+
```

If we now perform one dequeue operation,
```c
// Code Snippet 1-2
Dequeue()
```

only the element at the **front** is removed.
```bash
# Bash Snippet 1-2

Front                         Rear
  │                             │
  ▼                             ▼
+----+    +----+    +----+
| 20 | -> | 30 | -> | 40 |
+----+    +----+    +----+
```

Notice that although **40** was inserted last, it cannot leave the queue until every element before it has been removed.
## Queue Terminology
Every queue has two important pointers (or indices in an array implementation).
### Front
The **Front** refers to the first element currently stored inside the queue.
All dequeue operations occur from the front.
```bash
# Bash Snippet 1-3

Front
  │
  ▼
+----+    +----+    +----+
| 15 | -> | 20 | -> | 35 |
+----+    +----+    +----+
```

### Rear
The **Rear** refers to the last element currently stored inside the queue.
All enqueue operations occur at the rear.
```bash
# Bash Snippet 1-4

					 Rear
					   │
					   ▼
+----+    +----+    +----+
| 15 | -> | 20 | -> | 35 |
+----+    +----+    +----+
```
### Complete Queue
```bash
# Bash Snippet 1-5

Front                          Rear
  │                              │
  ▼                              ▼
+----+    +----+    +----+    +----+
| 15 | -> | 20 | -> | 35 | -> | 50 |
+----+    +----+    +----+    +----+
```

Every enqueue operation inserts a new element after the rear.

Every dequeue operation removes the front element.
### Types of Queues
There are four common types of queues.
- Simple Queue
- Circular Queue
- Priority Queue
- Double Ended Queue (Deque)

Each type is designed to solve a different problem.
#### 1. Simple Queue
The **Simple Queue** is the most basic implementation of a queue.
It follows the FIFO principle, where insertion always occurs at the rear and deletion always occurs from the front.
```bash
# Bash Snippet 1-6

Front                 Rear
  │                    │
  ▼                    ▼
+----+    +----+    +----+
| 10 | -> | 20 | -> | 30 |
+----+    +----+    +----+
```

```c
// Code Snippet 1-3

Enqueue(40)
```

```bash
# Bash Snippet 1-7

Front                          Rear
  │                             │
  ▼                             ▼
+----+    +----+    +----+    +----+
| 10 | -> | 20 | -> | 30 | -> | 40 |
+----+    +----+    +----+    +----+
```

Likewise,
```c
// Code Snippet 1-4

Dequeue()
```

```bash
# Bash Snippet 1-8

Front                Rear
  │                   │
  ▼                   ▼
+----+    +----+    +----+
| 20 | -> | 30 | -> | 40 |
+----+    +----+    +----+
```

Although this implementation is simple, it suffers from an important drawback when implemented using an array.

Consider an array of size **5**.
```bash
# Bash Snippet 1-9

Index

0    1    2    3    4
+----+----+----+----+----+
| 10 | 20 | 30 | 40 | 50 |
+----+----+----+----+----+
 F                     R
```

After three dequeue operations,
```bash
# Bash Snippet 1-10

0    1    2    3    4
+----+----+----+----+----+
| XX | XX | XX | 40 | 50 |
+----+----+----+----+----+
                 F    R
```

There are now three unused locations at the beginning of the array.
Unfortunately, if we enqueue another element,
```c
// Code Snippet 1-5

Enqueue(60)
```

the rear has already reached the last index.
```bash
# Bash Snippet 1-11

0    1    2    3    4
+----+----+----+----+----+
| XX | XX | XX | 40 | 50 |
+----+----+----+----+----+
                  F    R
```

Even though free locations exist, they cannot be reused.
The queue incorrectly reports **Queue Overflow**
This wasted memory is known as **False Overflow**.
To solve this problem, we use the **Circular Queue**.
#### 2. Circular Queue
A **Circular Queue** is an improved version of the simple queue.

Instead of treating the array as having a beginning and an end, the array is treated as though its last position is connected back to its first position, forming a circle.

```bash 
# Bash Snippet 1-12

+------------------------+
|                        |
▼                        ▲

+----+----+----+----+----+
|    |    |    |    |    |
+----+----+----+----+----+
```

Because of this circular arrangement, once the rear reaches the final array position, it wraps around to index **0**, provided that position is free.

Using the previous example,
```bash
# Bash Snippet 1-13

0    1    2    3    4
+----+----+----+----+----+
| XX | XX | XX | 40 | 50 |
+----+----+----+----+----+
                  F    R
```

Enqueueing another element results in

```bash
# Bash Snippet 1-14

0    1    2    3    4
+----+----+----+----+----+
| 60 | XX | XX | 40 | 50 |
+----+----+----+----+----+
 R               F
```

Instead of reporting overflow, the rear simply wraps around to the beginning of the array.
This eliminates false overflow while still allowing all queue operations to execute in **O(1)** time.

#### 3. Priority Queue
A **Priority Queue** does not necessarily remove elements in the order they were inserted.
Instead, each element is assigned a **priority**.
The element with the highest priority is removed first.

For example,
```c
// Code Snippet 1-6

Enqueue(A, Priority = 2)
Enqueue(B, Priority = 5)
Enqueue(C, Priority = 1)
Enqueue(D, Priority = 4)
```

The internal ordering becomes
```bash
# Bash Snippet 1-15

Highest Priority

+---------+---------+---------+---------+
| B (5)   | D (4)   | A (2)   | C (1)   |
+---------+---------+---------+---------+
```

Therefore,
```c
// Code Snippet 1-7

Dequeue()
```

returns
```bash
# Bash Snippet 1-16

B
```

instead of
```bash
# Bash Snippet 1-17

A
```

Priority queues are widely used in CPU scheduling, Dijkstra's shortest path algorithm, A* search, and operating systems.
#### 4. Double Ended Queue (Deque)
A **Double Ended Queue**, commonly abbreviated as **Deque**, allows insertion and deletion from **both ends** of the queue.

Unlike a simple queue,
- Elements may be inserted at either the front or rear.
- Elements may be removed from either the front or rear.
```bash
# Bash Snippet 1-18

Insert Front ◄───── Queue ─────► Insert Rear

Delete Front ◄──── Queue ─────► Delete Rear
```

A deque is useful in applications such as browser history, undo/redo systems, palindrome checking, and sliding window algorithms.

# Circular Array-based Queue
## CircularQueue.h
```c
// Code Snippet 2-1

#pragma once

#include "genericPrint.h"

#define SIZE 5

typedef struct {
	void *data;
	Datatype type;
} QueueData;

typedef struct {
	QueueData elements[SIZE];
	int frontIdx;
	int rearIdx;
	int size;
} Queue;

void initializeQueue(Queue *queue);

void enqueue(Queue *queue, void *data, Datatype type);
QueueData dequeue(Queue *queue);
QueueData peek(const Queue *queue);

int isFull(const Queue *queue);
int isEmpty(const Queue *queue);

int numberOfElements(const Queue *queue);
void displayElements(const Queue *queue);
void destroyQueue(Queue *queue);
```

Just as in the previous data structures, we begin by defining the data that will be stored inside the queue.
```c
// Code Snippet 2-2

typedef struct {
	void *data;
	Datatype type;
} QueueData;
```

Since we want the queue to be generic, every element stores both a pointer to the actual data and its datatype. This allows the queue to store values of any datatype while still being able to display them correctly using `genericPrint()`.

---

Next, we define the queue itself.
```c
// Code Snippet 2-3

typedef struct {
	QueueData elements[SIZE];
	int frontIdx;
	int rearIdx;
	int size;
} Queue;
```

Unlike the stack, a queue requires **two indices**.
The **front index** always refers to the first valid element in the queue. Every dequeue operation removes the element located at this index.
The **rear index** always refers to the last valid element in the queue. Every enqueue operation inserts the new element after this index.

Finally, we store the current number of elements in the queue using the member `size` which is the number of actual filled places of the `SIZE`.

Unlike a simple queue, the indices in a circular queue are allowed to wrap around the array. Whenever either index reaches the last position of the array, it returns to index **0**.

This behavior is accomplished using the **modulus operator (`%`)**.

---
## CircularQueue.c
```c
// Code Snippet 2-4

#include <stdio.h>
#include <stdlib.h>

#include "CircularQueue.h"

void initializeQueue(Queue *queue) {
	queue->frontIdx = 0;
	queue->rearIdx = -1;
	queue->size = 0;
}

int isFull(const Queue *queue) {
	return queue->size == SIZE;
}

int isEmpty(const Queue *queue) {
	return queue->size == 0;
}
```

The implementation requires `<stdio.h>` for displaying error messages and includes `"CircularQueue.h"` to access the queue declarations.

Unlike the linked list implementation, no dynamic memory allocation is performed since the queue stores its elements inside a fixed-size array.

### `void initializeQueue(Queue *queue);`
This method initializes the queue by assigning the default values to its members.

Initially, the queue contains no elements.
The front index is initialized to **0**, while the rear index is initialized to **-1** because no element has been inserted yet.
Finally, the size is initialized to **0**.
```c
// Code Snippet 2-5

void initializeQueue(Queue *queue) {
	queue->frontIdx = 0;
	queue->rearIdx = -1;
	queue->size = 0;
}
```

### `int isFull(const Queue *queue);`
This method returns whether the queue has reached its maximum capacity.

Unlike a stack, checking whether a circular queue is full cannot be done simply by examining the rear index, since both indices continuously wrap around the array.
Instead, we keep track of the current number of elements using the member `size`.
If the number of stored elements equals the array size, the queue is considered full.
```c
// Code Snippet 2-6

int isFull(const Queue *queue) {
	return queue->size == SIZE;
}
```

### `int isEmpty(const Queue *queue);`
This method returns whether the queue contains any elements.
If the size of the queue is zero, then the queue is empty.
```c
// Code Snippet 2-7

int isEmpty(const Queue *queue) {
	return queue->size == 0;
}
```

### `void enqueue(Queue *queue, void *data, Datatype type);`
This method inserts a new element at the rear of the circular queue.
First, the method checks whether the queue is already full. If every location in the array is occupied, no further insertion is possible, and a queue overflow message is displayed.
If space is available, the rear index advances to the next position using the modulus operator.
```c
// Code Snippet 2-8

queue->rearIdx = (queue->rearIdx + 1) % SIZE;
```

Unlike a simple queue, the rear index does not stop after reaching the final array position. Instead, whenever it reaches the end of the array, it wraps around to the beginning, allowing previously emptied locations to be reused.

Once the rear index has been updated, the entered data and datatype are stored at that position. Finally, the queue size is incremented to indicate that one more element has been inserted.
```c
// Code Snippet 2-9

void enqueue(Queue *queue, void *data, Datatype type) {
	if(isFull(queue)) {
		fprintf(stderr, "Queue Overflow!\n");
		return;
	}
	
	queue->rearIdx = (queue->rearIdx + 1) % SIZE;
	
	queue->elements[queue->rearIdx].data = data;
	queue->elements[queue->rearIdx].type = type;
	
	queue->size++;
}
```
#### Why `% SIZE`?
The modulus operator ensures that the rear index always remains within the valid range of the array.
Suppose the queue size is **5**, then the rear index changes as follows:

| Current Rear | Expression    | New Rear |
| ------------ | ------------- | -------- |
| 0            | `(0 + 1) % 5` | 1        |
| 1            | `(1 + 1) % 5` | 2        |
| 2            | `(2 + 1) % 5` | 3        |
| 3            | `(3 + 1) % 5` | 4        |
| 4            | `(4 + 1) % 5` | 0        |
Notice that after reaching the last index, the rear automatically returns to index **0**, making the array behave as though its beginning and end are connected.

### `QueueData dequeue(Queue *queue);`
This method removes and returns the element located at the front of the circular queue.

The method first checks whether the queue is empty. Removing an element from an empty queue is not possible, so a queue underflow message is displayed.

Otherwise, the element located at the current front index is copied into a temporary variable.
Next, the front index advances using the same circular movement employed by the rear index.
```c
// Code Snippet 2-10

queue->frontIdx = (queue->frontIdx + 1) % SIZE;
```

Finally, the queue size is decremented, and the removed element is returned.
Just like the rear index, the front index wraps around whenever it reaches the end of the array.
```c
// Code Snippet 2-11

QueueData dequeue(Queue *queue) {
	QueueData empty = {nullptr, TYPE_INT};
	
	if(isEmpty(queue)) {
		fprintf(stderr, "Queue Underflow!\n");
		return empty;
	}
	
	QueueData removed = queue->elements[queue->frontIdx];
	queue->frontIdx = (queue->frontIdx + 1) % SIZE;
	queue->size--;
	
	return removed;
}
```

### `QueueData peek(const Queue *queue);`
This method returns the element located at the front of the queue without removing it.

First, the method checks whether the queue is empty. If no elements exist, an error message is displayed and an empty `QueueData` object is returned.
Otherwise, the method simply returns the element located at the current front index.
Unlike `dequeue()`, this method does **not** modify the queue. Neither the front index nor the size changes.
```c
// Code Snippet 2-12

QueueData peek(const Queue *queue) {
	QueueData empty = {nullptr, TYPE_INT};
	
	if(isEmpty(queue)) {
		fprintf(stderr, "Queue is empty!\n");
		return empty;
	}
	
	return queue->elements[queue->frontIdx];
}
```
### `int numberOfElements(const Queue *queue);`
This method returns the current number of elements stored in the queue.

Since the queue maintains its size internally, there is no need to traverse the array or perform any calculations. The method simply returns the value of the `size` member.

As a result, this operation executes in **O(1)** time.
```c
// Code Snippet 2-13

int numberOfElements(const Queue *queue) {
	return queue->size;
}
```

### `void displayElements(const Queue *queue);`
This method displays every element currently stored in the queue, beginning from the front and ending at the rear.

First, it checks whether the queue is empty. If so, an error message is displayed.
Otherwise, a temporary index named `current` is initialized to the front index.

The method then iterates exactly `size` times. During each iteration, the current element is displayed using `genericPrint()`, after which the temporary index advances using the circular movement formula.
```c
// Code Snippet 2-14

current = (current + 1) % SIZE;
```
Using a temporary variable instead of the actual front index ensures that displaying the queue does not modify its contents.

Because the loop executes exactly `size` times, only the valid elements stored in the queue are displayed, regardless of how the indices are arranged internally.
```c
// Code Snippet 2-15

void displayElements(const Queue *queue) {
	if(isEmpty(queue)) {
		fprintf(stderr, "Queue is empty!\n");
		return;
	}
	
	int current = queue->frontIdx;
	
	for(int i = 0; i < queue->size; i++) {
		genericPrint(queue->elements[current].data);
		
		current = (current + 1) % SIZE;
	}
}
```

### `void destroyQueue(Queue *queue);`
Unlike the linked list implementation, a circular queue does **not** allocate memory dynamically. All elements are stored inside a fixed-size array that is allocated when the queue itself is created.

Therefore, there is no dynamically allocated memory to deallocate using `free()`.
Destroying the queue simply means resetting it to its initial empty state by calling `initializeQueue()`.

This resets the front index, rear index, and size, effectively removing all elements from the queue.
```c
// Code Snippet 2-16

void destroyQueue(Queue *queue) {
	initializeQueue(queue);
}
```

### Comparison with Other Queue Implementations

| Operation                     | Simple Queue | Circular Queue |
| ----------------------------- | ------------ | -------------- |
| Enqueue                       | O(1)         | O(1)           |
| Dequeue                       | O(1)         | O(1)           |
| Wasted Space (False Overflow) | **Yes**      | **No**         |
| Memory Utilization            | Poor         | Excellent      |
| Practical Usage               | Rare         | Very Common    |

The circular queue eliminates the **false overflow** problem found in the simple queue by treating the underlying array as circular. Through the use of the modulus operator, both the front and rear indices wrap around to the beginning of the array when they reach the end, allowing previously emptied locations to be reused efficiently. As a result, all primary queue operations execute in constant time while making full use of the available storage.

# Linked Queue
Above, we implemented the queue using a fixed-size circular array. Although this implementation efficiently utilizes the available array space, its maximum capacity remains limited by the array size.

An alternative implementation is the **Linked Queue**, where the queue is built using a **singly linked list** instead of an array.

Unlike the circular queue, a linked queue does not require a predefined size. Instead, every new element is dynamically allocated in memory using `malloc()`, allowing the queue to grow until the system runs out of available memory.

Despite the difference in implementation, the linked queue follows the same **FIFO (First In, First Out)** principle. Elements are always inserted at the **rear** of the queue and removed from the **front**.

To perform these operations efficiently, the linked queue maintains **two pointers**:
- **Front Pointer:** Points to the first node in the queue. Every dequeue operation removes this node.
- **Rear Pointer:** Points to the last node in the queue. Every enqueue operation inserts a new node after this node.

By maintaining both pointers, both insertion and deletion can be performed in constant time without traversing the linked list.
## Queue Structure
Initially, an empty linked queue contains no nodes.
```bash
# Bash Snippet 3-1

Front
 │
 ▼
NULL

Rear
 │
 ▼
NULL
```

After several enqueue operations,
```bash
# Bash Snippet 3-2

Front                                      Rear
 │                                          │
 ▼                                          ▼

+------+------+      +------+------+      +------+------+
| Data | Next | ───► | Data | Next | ───► | Data | NULL |
+------+------+      +------+------+      +------+------+
```

The **front pointer** always refers to the first node, while the **rear pointer** always refers to the last node.

Performing a dequeue operation removes the first node.
- Before:
```bash
# Bash Snippet 3-3


Front                                       Rear
 │                                           │
 ▼                                           ▼

+------+------+      +------+------+      +------+------+
| Data | Next | ───► | Data | Next | ───► | Data | NULL |
+------+------+      +------+------+      +------+------+
```

- After:
```Bash
# Bash Snippet 3-4

Front                  Rear
 │                      │
 ▼                      ▼

+------+------+      +------+------+
| Data | Next | ───► | Data | NULL |
+------+------+      +------+------+
```

Notice that only the **front pointer** changes after a dequeue operation.

## LinkedQueue.h
```c
// Code Snippet 3-1

#pragma once

#include "genericPrint.h"

typedef struct QueueNode {
	void *data;
	Datatype type;
	struct QueueNode *next;
} QueueNode;

typedef struct {
	QueueNode *frontNode;
	QueueNode *rearNode;
	int size;
} Queue;

void initializeQueue(Queue *queue);

void enqueue(Queue *queue, void *data, Datatype type);
QueueNode dequeue(Queue *queue);
QueueNode peek(const Queue *queue);

int isEmpty(const Queue *queue);
int numberOfElements(const Queue *queue);

void displayElements(const Queue *queue);
void destroyQueue(Queue *queue);
```

Unlike the circular queue, the linked queue does **not** require an array to store its elements.
Instead, every element is represented by a node that is dynamically allocated when inserted into the queue.

We therefore begin by defining the queue node.
```c
// Code Snippet 3-2

typedef struct QueueNode {
	void *data;
	Datatype type;
	struct QueueNode *next;
} QueueNode;
```

Just like the linked list implementation, every node stores three pieces of information.
- A generic pointer to the stored data.
- The datatype of the stored data.
- A pointer to the next node in the queue.

If the node is the last node in the queue, its `next` pointer stores `nullptr`.
Next, we define the queue itself.
```c
// Code Snippet 3-3

typedef struct {
	QueueNode *frontNode;
	QueueNode *rearNode;
	int size;
} Queue;
```

Unlike the linked list implementation, which only required a pointer to the head node, the linked queue stores **two node pointers**.

The first pointer, `frontNode`, always points to the first node in the queue. Since the queue follows the FIFO principle, every dequeue operation removes this node.
The second pointer, `rearNode`, always points to the last node in the queue. Every enqueue operation inserts a new node after this node.
Maintaining both pointers allows insertion and deletion to execute in **constant time**. If only one pointer were maintained, inserting a new node would require traversing the entire linked list to locate the last node, resulting in an **O(n)** operation.

Finally, the member `size` stores the current number of nodes in the queue, allowing the number of elements to be returned in constant time.

The function prototypes are very similar to those of the circular queue. However, the internal implementation differs significantly.
Instead of moving array indices using modular arithmetic, the linked queue manipulates node pointers.

Furthermore, since every node is dynamically allocated using `malloc()`, the implementation must deallocate nodes using `free()` when they are removed from the queue.
## LinkedQueue.c
```c
// Code Snippet 3-4

#include <stdio.h>
#include <stdlib.h>

#include "LinkedQueue.h"

void initializeQueue(Queue *queue) {
	queue->frontNode = nullptr;
	queue->rearNode = nullptr;
	queue->size = 0;
}

void enqueue(Queue *queue, void *data, Datatype type) {
	QueueNode *newNode = (QueueNode*) malloc(sizeof(QueueNode));
	
	if(newNode == nullptr) {
		fprintf(stderr, "Memory allocation failed!\n");
		return;
	}
	
	newNode->data = data;
	newNode->type = type;
	newNode->next = nullptr;
	
	if(isEmpty(queue)) {
		queue->frontNode = newNode;
		queue->rearNode = newNode;
	}
	else {
		queue->rearNode->next = newNode;
		queue->rearNode = newNode;
	}
	
	queue->size++;
}

QueueNode dequeue(Queue *queue) {
	QueueNode empty = {nullptr, TYPE_INT, nullptr};
	
	if(isEmpty(queue)) {
		fprintf(stderr, "Queue Underflow!\n");
		return empty;
	}
	
	QueueNode removed = *(queue->frontNode);
	
	QueueNode *deleteMe = queue->frontNode;
	queue->frontNode = queue->frontNode->next;
	
	if(queue->frontNode == nullptr) {
		queue->rearNode = nullptr;
	}
	
	free(deleteMe);
	
	queue->size--;
	
	return removed;
}

QueueNode peek(const Queue *queue) {
	QueueNode empty = {nullptr, TYPE_INT, nullptr};
	
	if(isEmpty(queue)) {
		fprintf(stderr, "Queue is empty!\n");
		return empty;
	}
	
	return *(queue->frontNode);
}

int isEmpty(const Queue *queue) {
	return (queue->frontNode == nullptr);
}

int numberOfElements(const Queue *queue) {
	return queue->size;
}

void displayElements(const Queue *queue) {
	if(isEmpty(queue)) {
		fprintf(stderr, "Queue is empty!\n");
		return;
	}
	
	QueueNode *current = queue->frontNode;
	
	while(current != nullptr) {
		genericPrint(current->data);
		current = current->next;
	}
}

void destroyQueue(Queue *queue) {
	while(!isEmpty(queue)) {
		dequeue(queue);
	}
}
```

Unlike the circular queue implementation, the linked queue relies on **dynamic memory allocation**. Therefore, the implementation includes `<stdlib.h>` to provide the functions `malloc()` and `free()`.
The header `<stdio.h>` is also included to display error messages whenever invalid queue operations are attempted.

### `void initializeQueue(Queue *queue);`
This method initializes an empty linked queue by assigning default values to all of its members.

Initially, the queue contains no nodes. Therefore, both the front and rear pointers point to `nullptr`, indicating that the queue is empty.

Finally, the queue size is initialized to zero.
```c
// Code Snippet 3-5

void initializeQueue(Queue *queue) {
	queue->frontNode = nullptr;
	queue->rearNode = nullptr;
	queue->size = 0;
}
```
### `void enqueue(Queue *queue, void *data, Datatype type);`
This method inserts a new element at the rear of the linked queue.

Since the queue is implemented using a linked list, the first step is to dynamically allocate memory for a new node using `malloc()`.
If memory allocation fails, the method displays an error message and terminates without modifying the queue.
Otherwise, the node is initialized by storing the provided data, its datatype, and assigning `nullptr` to its `next` pointer since it will become the last node in the queue.
The insertion process then depends on whether the queue is empty.

If the queue contains no nodes, the new node becomes both the **front** and **rear** of the queue.

```Bash
# Bash Snippet 3-5

Front ──► NULL

Rear  ──► NULL
```

- After
```bash
# Bash Snippet 3-6 

Front     Rear
 │         │
 ▼         ▼

+------+------+ 
| Data | NULL |
+------+------+
```

If the queue already contains nodes, the current rear node is linked to the new node, after which the rear pointer is updated to point to the newly inserted node.
- Before:
```bash
# Bash Snippet 3-7

Front                  Rear
 │                      │
 ▼                      ▼

+------+------+      +------+------+
| Data | Next | ───► | Data | NULL |
+------+------+      +------+------+
```

- After incrementing another node.
```c
// Code SNippet 3-6
queue->rearNode->next = newNode;
queue->rearNode = newNode;
```

```bash
# Bash Snippet 3-8

Front                                       Rear
 │                                           │
 ▼                                           ▼

+------+------+      +------+------+      +------+------+
| Data | Next | ───► | Data | Next | ───► | Data | NULL |
+------+------+      +------+------+      +------+------+
```

Finally, the queue size is incremented.
```c
// Code Snippet 3-7

void enqueue(Queue *queue, void *data, Datatype type) {
	QueueNode *newNode = (QueueNode*) malloc(sizeof(QueueNode));
	
	if(newNode == nullptr) {
		fprintf(stderr, "Memory allocation failed!\n");
		return;
	}
	
	newNode->data = data;
	newNode->type = type;
	newNode->next = nullptr;
	
	if(isEmpty(queue)) {
		queue->frontNode = newNode;
		queue->rearNode = newNode;
	}
	else {
		queue->rearNode->next = newNode;
		queue->rearNode = newNode;
	}
	
	queue->size++;
}
```
### `QueueNode dequeue(Queue *queue);`
This method removes and returns the node located at the front of the queue.

First, the method checks whether the queue is empty. Attempting to remove an element from an empty queue is invalid, so an error message is displayed and an empty node is returned.
Otherwise, the contents of the front node are copied into a temporary variable named `removed`.
A temporary pointer named `deleteMe` is then used to remember the node that will be removed.

Next, the front pointer advances to the following node.
```c
// Code Snippet 3-8

queue->frontNode = queue->frontNode->next;
```

If the removed node was the **only node** in the queue, the front pointer becomes `nullptr`. In this case, the rear pointer must also be updated to `nullptr` to correctly indicate that the queue is now empty.
- Before:
```Bash
# Bash Snippet 3-9

Front      Rear
 │          │
 ▼          ▼

+------+------+
| Data | NULL |
+------+------+
```

After removing the node,
```Bash
# Bash Snippet 3-10

Front ──► NULL

Rear  ──► NULL
```

If this update were omitted, the rear pointer would continue pointing to memory that had already been deallocated, creating a **dangling pointer**.

Finally, the removed node is deallocated using `free()`, the queue size is decremented, and the copied node is returned.
```c
// Code Snippet 3-9

QueueNode dequeue(Queue *queue) {
	QueueNode empty = {nullptr, TYPE_INT, nullptr};
	
	if(isEmpty(queue)) {
		fprintf(stderr, "Queue Underflow!\n");
		return empty;
	}
	
	QueueNode removed = *(queue->frontNode);
	
	QueueNode *deleteMe = queue->frontNode;
	queue->frontNode = queue->frontNode->next;
	
	if(queue->frontNode == nullptr) {
		queue->rearNode = nullptr;
	}
	
	free(deleteMe);
	
	queue->size--;
	
	return removed;
}
```

### `QueueNode peek(const Queue *queue);`
This method returns the node located at the front of the queue without removing it.

First, the method checks whether the queue is empty. If the queue contains no nodes, an error message is displayed and an empty node is returned.
Otherwise, the contents of the front node are returned directly.

Unlike `dequeue()`, this method does **not** modify the queue. The front pointer, rear pointer, and queue size all remain unchanged.
```c
// Code Snippet 3-10

QueueNode peek(const Queue *queue) {
	QueueNode empty = {nullptr, TYPE_INT, nullptr};
	
	if(isEmpty(queue)) {
		fprintf(stderr, "Queue is empty!\n");
		return empty;
	}
	
	return *(queue->frontNode);
}
```

### `int isEmpty(const Queue *queue);`
This method returns whether the queue is empty.

Since an empty linked queue contains no nodes, the front pointer points to `nullptr`.
If the front pointer is `nullptr`, the queue is empty and the method returns `1`. Otherwise, it returns `0`.
```c
// Code Snippet 3-11

int isEmpty(const Queue *queue) {
	return (queue->frontNode == nullptr);
}
```

### `int numberOfElements(const Queue *queue);`
This method returns the current number of nodes stored in the queue.

Since the queue maintains its size internally, there is no need to traverse the linked list. The method simply returns the value stored in the `size` member.

As a result, this operation executes in **O(1)** time.
```c
// Code Snippet 3-12

int numberOfElements(const Queue *queue) {
	return queue->size;
}
```
### `void displayElements(const Queue *queue);`
This method displays every element currently stored in the queue.

First, the method checks whether the queue is empty. If so, an error message is displayed.
Otherwise, a temporary pointer named `current` is initialized to the front node.
The method then traverses the queue one node at a time until the end of the linked list is reached.

During each iteration, the current node's data is displayed using `genericPrint()`. The temporary pointer is then advanced to the following node.
```c
// Code Snippet 3-13

current = current->next;
```
The traversal terminates when `current` becomes `nullptr`, indicating that every node has been visited.

Using a temporary pointer ensures that the queue itself remains unchanged while its elements are displayed.
```c
// Code Snippet 3-14

void displayElements(const Queue *queue) {
	if(isEmpty(queue)) {
		fprintf(stderr, "Queue is empty!\n");
		return;
	}
	
	QueueNode *current = queue->frontNode;
	
	while(current != nullptr) {
		genericPrint(current->data, current->type);
		current = current->next;
	}
}
```

### `void destroyQueue(Queue *queue);`
This method deallocates every node currently stored in the queue.

Unlike the circular queue implementation, every node in a linked queue is dynamically allocated using `malloc()`. Therefore, every node must eventually be deallocated using `free()` to prevent memory leaks.

Rather than manually traversing the queue and freeing each node, this method repeatedly removes the front node by calling `dequeue()`.

Each call to `dequeue()` performs the following operations:
- Removes the front node.
- Advances the front pointer to the next node.
- Deallocates the removed node using `free()`.
- Updates the queue size.

The loop continues until the queue becomes empty.
```c
// Code Snippet 3-15

void destroyQueue(Queue *queue) {
	while(!isEmpty(queue)) {
		dequeue(queue);
	}
}
```

# Linked-Based Queue vs Circular Array-Based Queue

|Feature|Circular Queue|Linked Queue|
|---|---|---|
|Underlying Structure|Array|Singly Linked List|
|Maximum Capacity|Fixed|Dynamic|
|Memory Allocation|Static|Dynamic (`malloc()`)|
|False Overflow|No|No|
|Enqueue|**O(1)**|**O(1)**|
|Dequeue|**O(1)**|**O(1)**|
|Memory Overhead|Lower|Higher (one pointer per node)|
|Cache Performance|Better|Worse|
|Memory Utilization|Fixed-size array|Allocates memory as needed|

The circular queue and linked queue both provide constant-time enqueue and dequeue operations. However, they differ in how they manage memory.

A circular queue stores its elements inside a fixed-size array, making it memory-efficient and cache-friendly. Its maximum capacity is determined when the queue is created.

In contrast, a linked queue stores each element in a dynamically allocated node. This allows the queue to grow and shrink as needed without a predefined limit. The trade-off is that each node requires additional memory for its pointer, and dynamic memory allocation introduces a small performance overhead.

For applications with a known maximum capacity, a circular queue is often the better choice. When the required capacity is unknown or may vary significantly during execution, a linked queue provides greater flexibility.
